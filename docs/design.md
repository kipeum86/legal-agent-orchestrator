## Problem Statement

10개 전문 법률 AI 에이전트가 독립적으로 존재하지만, 이들을 통합적으로 조율하는 레이어가 없다. 클라이언트(사용자)가 법률 질문을 던지면 적절한 에이전트에게 배정하고, 에이전트 간 협업을 조율하고, 최종 결과물을 전달하는 "로펌" 역할의 오케스트레이터를 구축한다.

**핵심 원칙: 결과물 퀄리티 > 시각적 임팩트.** 기존 에이전트의 풀 스택(MCP, 스킬, KB, 팩트체커)을 100% 유지한다. 웹 래퍼로 기능을 깎는 것은 의미 없다.

## What Makes This Cool

**"보이지 않는 것을 보이게 만든다."**

Harvey AI는 블랙박스다. 법무법인 진주는 정반대:
- 어떤 변호사가 배정되었는지
- 어떤 소스(Grade A/B/C)를 참조했는지
- 팩트체크에서 뭐가 걸렸는지
- 에이전트 간 법률 논쟁(멀티라운드 토론)이 어떻게 진행되었는지
- 파트너 검토에서 어떤 코멘트가 달렸는지

**전부 보인다.** 기존 에이전트를 100% 그대로 쓰기 때문에, 실제 법률 전문가가 쓸 수 있는 수준의 결과물이 나온다.

**핵심 차별화 — 멀티라운드 토론:**
- 서로 다른 관할권의 전문 변호사가 의견을 주고받으며 논쟁
- PIPA-expert가 한국법 의견을 내고, GDPR-expert가 EU법 관점에서 반론하고, 재반론이 오가고, 파트너가 최종 판단
- 이건 단일 LLM 호출로는 절대 만들 수 없는 깊이. 각 에이전트가 자체 KB와 소스 그레이딩을 가지고 있기 때문에 가능.

**Case Report:** 처리된 케이스의 전체 협업 과정을 하나의 `case-report.md`로 묶어 GitHub에서 바로 읽히는 단일 아카이브로 남긴다. 포트폴리오 방문자는 폴더를 뒤지지 않고도 사건 전체를 한 파일에서 파악할 수 있다.

## Constraints

- **기존 에이전트 100% 재활용**: CLAUDE.md, skills, KB, MCP 설정 전부 그대로 사용
- **Claude Code 네이티브**: 오케스트레이터 실행에 웹 서버 불필요, 유저가 로컬에서 실행. 결과 전달도 웹 앱이 아니라 케이스 폴더 내부 산출물로 끝난다.
- **결과물 퀄리티 최우선**: 전달 형식은 단일 `case-report.md`로 단순화하고, 실행 환경을 타협하지 않음
- **오픈소스 포트폴리오**: Apache License 2.0, GitHub 공개

## Premises

1. **기존 에이전트를 100% 재활용한다** — CLAUDE.md, skills, KB, MCP 설정 전부 그대로. 어댑터 레이어 최소화.
2. **Claude Code가 실행 환경이다** — Agent SDK 웹서비스가 아니라 Claude Code CLI/Desktop/VS Code에서 실행.
3. **오케스트레이터는 라우팅 + 조율 레이어** — 질문 분류 → 에이전트 배정 → 파이프라인 관리 → 결과 조합.
4. **최종 전달은 `case-report.md`로 수렴** — 실행은 Claude Code, 전달은 케이스 폴더 내부 단일 Markdown 아카이브.
5. **에이전트 관리는 setup.sh** — git submodule이 아닌 setup 스크립트로 에이전트 클론/업데이트. 유지보수 단순화.

## Approaches Considered

### Approach A: Agent SDK 웹서비스 (기각)
Next.js + FastAPI + Agent SDK. 워룸 시각화는 화려하지만, 에이전트 기능의 50-60%만 재현. MCP 통합 불확실. **결과물 퀄리티 타협 불가.**

### Approach B: Claude Code 네이티브 + Single-File Case Report (채택)
오케스트레이터를 Claude Code 프로젝트로 구축. 기존 에이전트 100% 재활용. 최종 전달은 `case-report.md` 단일 산출물로 정리.

### Approach C: 하이브리드 (후속 고려)
B를 먼저 완성한 후, 라이트 웹 서비스를 선택적으로 추가.

### setup.sh — 에이전트 관리 스크립트

```bash
#!/bin/bash
# setup.sh — 에이전트 자동 클론 및 업데이트
# 사용법: ./setup.sh          (전체 클론/업데이트)
#         ./setup.sh update    (업데이트만)
#         ./setup.sh status    (각 에이전트 상태 확인)

GITHUB_USER="kipeum86"
AGENTS_DIR="agents"

# 에이전트 목록 — 새 에이전트 추가 시 여기에 한 줄 추가
REPOS=(
  "general-legal-research"
  "legal-writing-agent"
  "second-review-agent"
  "GDPR-expert"
  "PIPA-expert"
  "game-legal-research"
  "contract-review-agent"
  "legal-translation-agent"
  "game-legal-briefing"
  "game-policy-briefing"
)

mkdir -p "$AGENTS_DIR"

case "${1:-setup}" in
  setup|update)
    for repo in "${REPOS[@]}"; do
      if [ -d "$AGENTS_DIR/$repo" ]; then
        echo "📥 Updating $repo..."
        (cd "$AGENTS_DIR/$repo" && git pull --rebase)
      else
        echo "📦 Cloning $repo..."
        git clone "https://github.com/$GITHUB_USER/$repo" "$AGENTS_DIR/$repo"
      fi
    done
    echo "✅ All agents ready."
    ;;
  status)
    for repo in "${REPOS[@]}"; do
      if [ -d "$AGENTS_DIR/$repo" ]; then
        BRANCH=$(cd "$AGENTS_DIR/$repo" && git branch --show-current)
        COMMIT=$(cd "$AGENTS_DIR/$repo" && git log -1 --format="%h %s" 2>/dev/null)
        echo "✅ $repo ($BRANCH): $COMMIT"
      else
        echo "❌ $repo: not cloned"
      fi
    done
    ;;
esac
```

**새 에이전트 추가 시:**
1. `REPOS` 배열에 레포 이름 한 줄 추가
2. `./setup.sh` 실행
3. 오케스트레이터의 `route-case.md`에 라우팅 규칙 추가

**기존 에이전트 업데이트 시:**
- `./setup.sh update` — 모든 에이전트 최신화 (git pull)
- 에이전트 레포에서 작업 → push → 오케스트레이터에서 setup.sh 한 번이면 끝

### Routing Logic

오케스트레이터의 `route-case.md` 스킬이 질문을 분류하고 에이전트 조합을 결정:

```
질문 입력
  ├─ 데이터보호 + EU       → GDPR-expert → legal-writing → second-review
  ├─ 데이터보호 + 한국      → PIPA-expert → legal-writing → second-review
  ├─ 게임규제              → game-legal-research → legal-writing → second-review
  ├─ 계약 관련             → contract-review → second-review
  ├─ 번역 요청             → legal-translation (단독)
  ├─ 일반 법률 질문         → general-legal-research → legal-writing → second-review
  ├─ 복합 (한국+EU 비교)    → [PIPA-expert ∥ GDPR-expert] → legal-writing → second-review
  ├─ cross-jurisdiction 토론 → Pattern 3 (멀티라운드)
  ├─ 게임규제 뉴스/브리핑    → game-legal-briefing 또는 game-policy-briefing (단독, 모니터링용)
  └─ 분류 모호             → general-legal-research (기본 라우트)
                             또는 사용자에게 "어느 분야에 초점을 맞출까요?" 질문
```

**분류 기준:**
- 법역(jurisdiction): KR, EU, US, international, multi
- 분야(domain): data_protection, game_regulation, contract, general, translation
- 작업유형(task): research, drafting, review, translation, briefing, debate
- 복잡도(complexity): simple(에이전트 1개), compound(파이프라인), multi_domain(병렬), adversarial(토론)

**분류 방식:** `route-case.md`는 LLM 기반 분류 프롬프트. 사용자 질문을 분석하여 위 4개 차원을 JSON으로 추출. few-shot 예시 5-10개를 프롬프트에 포함하여 분류 일관성 확보. 예:
```
입력: "한국과 EU의 개인정보 국외이전 규제를 비교 분석해줘"
→ { "jurisdiction": ["KR", "EU"], "domain": "data_protection", "task": "research", "complexity": "multi_domain" }
→ Pattern 1: [PIPA-expert ∥ GDPR-expert] → legal-writing → second-review
```

**Pattern 1 vs Pattern 3 구분 기준:**
- **Pattern 1 (독립 리서치 → 통합)**: 사용자가 비교, 차이점, 현황 분석을 요청. 각 에이전트가 독립적으로 리서치하고 writing이 종합. 예: "한국과 EU의 국외이전 규제 비교"
- **Pattern 3 (멀티라운드 토론)**: 사용자가 명시적으로 양측 의견/논쟁/토론을 요청하거나, 두 관할권의 법적 입장이 **상충(conflicting)**할 때 선택. 핵심: 에이전트 A의 결론을 에이전트 B가 반박. 예: "한-EU 적정성 결정의 법적 효력 차이에 대해 양측 의견을 들려줘"
- **트리거 키워드**: "양측 의견", "논쟁", "반론", "토론", "상충", "갈등", "debate" → Pattern 3

---

### Collaboration Patterns (에이전트 협업 패턴)

**Pattern 1: 독립 리서치 → 통합**
동일 쟁점을 다른 관할권에서 독립 분석 (병렬 가능 시 병렬, 불가 시 순차). 결과를 legal-writing이 통합.
```
오케스트레이터 → [PIPA-expert ∥ GDPR-expert] → legal-writing → second-review
```
예: "한국과 EU의 개인정보 국외이전 규제 비교"

**Pattern 2: 순차 핸드오프**
한쪽의 분석 결과를 다른 에이전트가 참조하여 추가 분석.
```
오케스트레이터 → GDPR-expert → game-legal-research(GDPR 결과 참조) → legal-writing → second-review
```
예: "한국 게임사가 EU에 서비스할 때 loot box 컴플라이언스"

**Pattern 3: 멀티라운드 토론 (핵심 차별화)**
서로 다른 관할권/전문 분야의 에이전트가 의견을 주고받으며 논쟁. 실제 로펌에서 파트너들이 쟁점을 토론하는 과정을 재현.
```
오케스트레이터 → Agent A: "의견 제시"                → 결과 A
             → Agent B: "결과 A에 대해 반론/보충"     → 결과 B
             → Agent A: "결과 B에 대해 재반론"        → 결과 C
             → legal-writing: "A, B, C 종합 verdict"  → 최종 의견서
             → second-review: 파트너 최종 판단
```
예: "개인정보 국외이전에서 한국 적정성 결정과 EU 적정성 결정의 법적 효력 차이 — 양측 의견을 들려줘"

**라운드 제한:** 토론은 최대 3라운드 (의견 → 반론 → 재반론). 무한 루프 방지.
**이벤트 로깅:** 각 라운드가 events.jsonl에 기록되어, 이후 `case-report.md`에서 토론 과정을 서사형으로 재구성할 수 있다.

### Sub-Agent Invocation

**"Agent tool"이란:** Claude Code의 내장 기능으로, 서브 대화를 지정된 working directory에서 생성하는 도구. `Agent(prompt="...", cwd="agents/GDPR-expert/")`로 호출하면 해당 디렉토리의 CLAUDE.md를 시스템 프롬프트로 로드한 별도의 Claude 세션이 생성되고, 작업 완료 후 결과를 부모 세션에 반환한다. 각 서브에이전트는 독립 컨텍스트 윈도우를 가진다.

오케스트레이터가 하위 에이전트를 호출하는 방식 (Phase 0에서 결정):

**Option A: Claude Code Agent tool (우선 검토)**
```
오케스트레이터 CLAUDE.md에서 Agent tool로 서브에이전트 디스패치:
- working directory를 해당 에이전트 폴더로 지정 (예: agents/GDPR-expert/)
- 에이전트의 CLAUDE.md가 자동으로 로드됨
- 해당 에이전트의 skills, KB, MCP 설정 전부 활성화
- 결과를 오케스트레이터에 반환
```
장점: 기존 에이전트 100% 재활용, 추가 어댑터 불필요
검증 필요: Agent tool의 cwd 지정 시 CLAUDE.md 자동 로드 여부

**Option B: Claude Agent SDK (프로그래매틱, 백업)**
```python
from claude_agent_sdk import Agent
research = Agent(cwd="agents/general-legal-research/")
result = research.run("한국 게임규제 현황 리서치")
```
장점: 프로그래매틱 제어, 이벤트 스트리밍 직접 제어 가능
단점: 어댑터 레이어 필요, MCP 연결 검증 필요

**Phase 0에서 Option A 먼저 검증. 실패 시 Option B로 전환.**

**Fallback C (Option A, B 모두 실패 시) — 기능 열화 매트릭스:**
에이전트 CLAUDE.md에서 핵심 워크플로우만 추출한 "agent profile" (5-10K tokens)을 프롬프트에 주입.

| 기능 | Option A/B | Fallback C |
|------|-----------|------------|
| CLAUDE.md 시스템 프롬프트 | 자동 로드 | 핵심 발췌 주입 (축약) |
| skills/ 스킬 | 자동 인식 | 주요 스킬 1-2개만 프롬프트에 포함 |
| MCP 도구 (korean-law 등) | 접근 가능 | **접근 불가** — 웹 검색 fallback |
| knowledge/ KB 파일 브라우징 | Read/Grep/Glob 가능 | **접근 불가** — 요약된 참조만 |
| 소스 그레이딩 | 풀 A/B/C/D 시스템 | 제한적 (KB 없이 그레이딩 불완전) |
| 팩트체커 서브에이전트 | 동작 | **동작 불가** |

**Fallback C는 "100% 재활용" 전제를 위반.** 이 경우 프로젝트의 핵심 가치가 훼손되므로 **go/no-go 결정 지점**이다. Fallback C로 가야 한다면, 프로젝트 방향 자체를 재검토한다.

### Agent Output Contract

서브에이전트의 출력을 오케스트레이터가 안정적으로 소비하기 위한 계약:

**방식:** 오케스트레이터가 서브에이전트를 호출할 때, 프롬프트 끝에 출력 형식 지시를 포함한다.

```
[오케스트레이터 → 서브에이전트 호출 프롬프트 예시]

"한국 PIPA 관점에서 개인정보 국외이전 규제를 분석해주세요.

작업 완료 후 반드시 다음 JSON 블록을 응답 마지막에 포함하세요:
```json
{
  "summary": "2000 tokens 이내 핵심 요약",
  "key_findings": ["발견 1", "발견 2"],
  "sources": [{"title": "법률명", "grade": "A/B/C", "citation": "조문"}],
  "full_output_path": "output/{case-id}/pipa-research.md"
}
```
전체 분석 결과는 output/{case-id}/pipa-research.md에 저장하세요."
```

**기존 에이전트 수정 불필요:** 이 출력 형식 지시는 오케스트레이터의 프롬프트에만 추가. 에이전트의 CLAUDE.md를 변경하지 않는다. Claude는 프롬프트의 출력 형식 지시를 따른다.

**Fallback 파서:** 서브에이전트가 JSON 블록 없이 자연어만 반환할 경우, 오케스트레이터가 반환된 텍스트에서 요약을 직접 생성한다 (추가 LLM 호출 없이 오케스트레이터 자체가 요약).

**Phase 1 검증 기준:** JSON 블록 반환 성공률을 측정. 80% 미만이면 프롬프트 강화 (XML 태그 래핑, 예시 추가) 또는 응답 후 별도 structured extraction 호출 검토.

### Handoff Protocol (에이전트 간 데이터 전달)

```
[Stage 1: 오케스트레이터 → Research Agent]
프롬프트: "질문: {user_query}. 결과를 output/{case-id}/research.md에 저장하고 JSON 블록으로 요약 반환."
→ Agent 반환: { summary, key_findings, sources, full_output_path }

[Stage 2: 오케스트레이터 → Writing Agent]
프롬프트: "다음 리서치 요약을 바탕으로 법률 의견서를 작성하세요.
[리서치 요약]: {summary}
[주요 발견]: {key_findings}
[전체 리서치 참조 필요 시]: output/{case-id}/research.md를 Read하세요.
결과를 output/{case-id}/opinion.md에 저장하고 JSON 블록으로 요약 반환."

[Stage 3: 오케스트레이터 → Review Agent]
프롬프트: "다음 의견서를 파트너 검토하세요.
[의견서 경로]: output/{case-id}/opinion.md를 Read하세요.
검토 결과를 JSON 블록으로 반환."
```

핵심: 오케스트레이터가 **명시적으로 파일 경로와 요약을 프롬프트에 삽입**하여 다음 에이전트에 전달. 에이전트 간 직접 통신 없음 — 오케스트레이터가 모든 데이터 흐름을 중재.

### Event Emission Mechanism

`emit-events.md` 스킬의 구체적 동작:

```
[오케스트레이터 내부 프로세스]

1. 서브에이전트 호출 전:
   → Bash tool로 events.jsonl에 agent_assigned 이벤트 append
   echo '{"id":"evt_N","ts":"...","agent":"GDPR-expert","type":"agent_assigned","data":{...}}' >> output/{case-id}/events.jsonl

2. 서브에이전트 호출 → 결과 반환

3. 서브에이전트 반환 후:
   → 반환된 JSON 블록에서 sources, key_findings 파싱
   → 각 source에 대해 source_graded 이벤트 생성
   → Bash tool로 events.jsonl에 append

4. 에러 발생 시:
   → error 이벤트를 events.jsonl에 append
```

**case-id 생성:** 오케스트레이터가 사건 접수 시 `{YYYYMMDD}-{HHmmss}-{4자리 해시}` 형식으로 생성 (예: `20260402-103000-a7f2`). `route-case.md` 스킬에서 첫 번째 단계로 실행.

### Event Schema (정형화)

모든 이벤트는 `output/{case-id}/events.jsonl`에 JSONL 형식으로 기록.

**공통 envelope:**
```json
{
  "id": "evt_{sequence}",
  "ts": "ISO-8601 timestamp",
  "agent": "agent-id 또는 orchestrator",
  "type": "event_type",
  "data": { ... }
}
```

**이벤트 타입별 data 필드:**

| type | required data fields | description |
|------|---------------------|-------------|
| `case_received` | `query`, `jurisdiction[]`, `domain`, `complexity`, `pipeline[]` | 사건 접수 + 분류 결과 |
| `agent_assigned` | `agent_id`, `name`, `role`, `reason` | 담당 변호사 배정 |
| `research_started` | `agent_id`, `search_strategy` | 리서치 시작 |
| `source_found` | `agent_id`, `source_title`, `source_type` | 소스 발견 |
| `source_graded` | `agent_id`, `source`, `grade` (A/B/C/D), `relevance` | 소스 등급 판정 |
| `fact_checked` | `agent_id`, `claim`, `status` (verified/unverified/contradicted), `evidence` | 팩트체크 결과 |
| `debate_round` | `agent_id`, `round` (1/2/3), `position` (opinion/rebuttal/surrebuttal), `summary` | 멀티라운드 토론 |
| `draft_started` | `agent_id`, `document_type`, `sections[]` | 문서 작성 시작 |
| `draft_section_complete` | `agent_id`, `section_name`, `word_count` | 섹션 작성 완료 |
| `review_started` | `agent_id`, `reviewer_name`, `review_intensity` | 파트너 검토 시작 |
| `review_comment` | `agent_id`, `section`, `severity` (critical/major/minor), `comment` | 검토 코멘트 |
| `review_approved` | `agent_id`, `final_grade`, `conditions` | 최종 승인 |
| `debate_initiated` | `topic`, `participants[]`, `max_rounds` | 멀티라운드 토론 시작 |
| `debate_concluded` | `verdict_summary`, `consensus_areas[]`, `disagreement_areas[]` | 토론 종결 + verdict |
| `error` | `agent_id`, `error_type` (timeout/failure/mcp_unavailable), `message`, `attempt`, `max_attempts` | 오류 발생 (재시도 추적 포함) |
| `user_prompt` | `question`, `options[]`, `context` | 사용자 확인 필요 (라우팅 모호, 추가 정보 요청 등) |
| `user_response` | `answer`, `selected_option` | 사용자 응답 수신 |
| `pipeline_aborted` | `reason`, `last_completed_step`, `partial_output_path` | 파이프라인 중단 |
| `final_output` | `file_path`, `format`, `summary`, `total_sources`, `grade_distribution` | 최종 결과물 |

**파이프라인 일시 정지/재개:** 라우팅 모호 등으로 사용자 입력이 필요한 경우, 오케스트레이터가 `user_prompt` 이벤트를 기록하고 Claude Code의 자연스러운 대화 흐름으로 사용자에게 질문. 응답 후 `user_response` 기록하고 파이프라인 재개. Claude Code 환경이므로 별도 pause/resume 메커니즘 불필요 — 대화 자체가 pause/resume.

**멀티라운드 토론 이벤트 예시:**
```json
{"id":"evt_010","ts":"2026-04-02T10:30:00Z","agent":"PIPA-expert","type":"debate_round","data":{"round":1,"position":"opinion","summary":"한국 PIPA 제28조의2에 따르면 적정성 평가는 개인정보보호위원회 고시에 의해..."}}
{"id":"evt_011","ts":"2026-04-02T10:32:00Z","agent":"GDPR-expert","type":"debate_round","data":{"round":2,"position":"rebuttal","summary":"EU GDPR 제45조의 적정성 결정은 EU Commission이 직접 심사하며, 한국 방식과 근본적 차이는..."}}
{"id":"evt_012","ts":"2026-04-02T10:34:00Z","agent":"PIPA-expert","type":"debate_round","data":{"round":3,"position":"surrebuttal","summary":"반론 인정하나, PIPC 고시 2024-1호에 따르면 한국도 상호주의 원칙을 도입하여..."}}
```

### Single-File Case Report

`output/{case-id}/case-report.md` 또는 `samples/{case-id}/case-report.md` — GitHub에서 바로 읽히는 정적 Markdown 아카이브:

**Phase 3 MVP 기능:**

- `opinion.md` 본문 인라인 삽입

- [ ] **서브에이전트 호출 검증** — Agent tool로 하위 에이전트 호출. 다음 5개 기준 전부 통과해야 함:

  - 실패 시: Option B (Agent SDK) 검토. Option B도 실패 시: 에이전트의 CLAUDE.md에서 핵심 워크플로우만 추출한 "agent profile" (5-10K tokens)을 프롬프트에 주입하는 방식. (전체 CLAUDE.md 주입은 토큰 비용 과다로 비현실적)

## Token Budget & Context Management

각 에이전트의 CLAUDE.md + skills + KB 참조가 Claude의 컨텍스트 윈도우를 상당량 소비.

**컨텍스트 격리 모델:** Agent tool로 서브에이전트를 호출하면 별도의 컨텍스트 윈도우가 생성됨 (오케스트레이터와 분리). 따라서 토큰 예산은 "오케스트레이터 컨텍스트" + "각 서브에이전트 독립 컨텍스트"로 관리.

**전략: 구조화된 결과 반환 (Structured Hand-off)**
- 서브에이전트는 작업 후 구조화된 결과를 반환:
  ```json
  {
    "summary": "< 2000 tokens 이내의 핵심 요약",
    "key_findings": ["발견 1", "발견 2", ...],
    "sources": [{"title": "...", "grade": "A", "citation": "..."}],
    "full_output_path": "output/{case-id}/research-result.md"
  }
  ```
- 오케스트레이터 → 다음 서브에이전트: `summary` + `key_findings`만 전달 (전체 출력 X)
- 전체 결과물은 파일로 저장, 필요 시 다음 에이전트가 파일 직접 Read
- 멀티라운드 토론: 이전 라운드의 summary (< 2000 tokens)를 다음 라운드에 전달

**예상 토큰 사용량:**

| 파이프라인 | 에이전트 | 예상 토큰 | 예상 비용 (Sonnet) |
|-----------|---------|----------|-------------------|
| 단순 리서치 | 1 research + 1 writing + 1 review | ~100-180K | $1-3 |
| 독립 비교 | 2 research + 1 writing + 1 review | ~150-250K | $2-4 |
| 멀티라운드 토론 (낙관) | 2 research (3 rounds) + 1 writing + 1 review | ~250-400K | $4-8 |
| 멀티라운드 토론 (비관) | KB 집중 탐색 시 라운드당 ~100K | ~400-700K | $7-12 |

**모델 가정:** 모든 에이전트는 Claude Sonnet 4.6 사용 (기본값). 비용 계산: input $3/MTok, output $15/MTok 기준. 사용자가 Opus 사용 시 비용 3-4배 증가.

```
jinju-orchestrator/
├── .env.example          ← 필요한 환경변수 목록 (값 없이, git 포함)
├── .env                  ← 실제 키 (gitignore)

    └── 각 에이전트/       ← 부모 .env를 참조
```

| 환경변수 | 설명 | 필요한 에이전트 | 필수 여부 |
|---------|------|--------------|---------|
| `ANTHROPIC_API_KEY` | Anthropic API 인증 | 모든 에이전트 | **필수** |
| `KOREAN_LAW_API_KEY` | law.go.kr Open Law API | general-legal-research, PIPA-expert, game-legal-research | 선택 (없으면 웹 검색 fallback) |
| `KOREAN_LAW_MCP_HOST` | korean-law MCP 서버 주소 | general-legal-research, PIPA-expert | 선택 (MCP 미연결 시 웹 검색) |

## Error Handling

| 상황 | 타임아웃 | 처리 |
|------|---------|------|
| 리서치 에이전트 | 5분 (P95 관측치 ~3분의 1.5배) | 부분 결과 수용 + `error` 이벤트 기록. 재시도 1회 (아래 참조). |
| 작성 에이전트 | 3분 (P95 ~2분의 1.5배) | 실패 시 파이프라인 중단 (리서치 없이 작성 불가). |
| 검토 에이전트 | 2분 (P95 ~1분의 2배) | 경고와 함께 결과 전달 ("파트너 검토 미완료" 표시). |
| MCP 서버 미연결 | — | fallback: MCP 없이 웹 검색으로 리서치 가능 |
| 라우팅 모호 | — | `general-legal-research`를 기본 라우트로 사용. 또는 사용자에게 확인 질문. |
| 토론 교착 | 3라운드 | 강제 종료, 현재까지의 논점을 legal-writing에 전달. |
| 파이프라인 부분 실패 | — | 완료된 단계의 output은 보존 (output/{case-id}/에 저장됨). 재개(resume)는 MVP 범위 밖 — 사용자가 재실행. Phase 2+에서 `pipeline_aborted` 이벤트의 `last_completed_step`을 읽어 재개하는 기능 검토. |

**재시도(Retry) 정책:**
- 재시도 대상: 리서치 에이전트만 (작성/검토는 재시도 없이 중단 또는 경고)
- 재시도 횟수: 최대 1회
- 재시도 프롬프트: 동일 프롬프트 재사용 (LLM 비결정성으로 다른 결과 가능)
- 재시도 타임아웃: 원래와 동일 (5분)
- 재시도 비용: 케이스당 ~$1-3 추가. 빈도가 높으면(>20%) 프롬프트 또는 에이전트 설정 점검
- 이벤트 기록: 재시도 시 `error` 이벤트에 `attempt: 2, max_attempts: 2` 기록

## Cross-Model Perspective

Claude 서브에이전트의 독립 평가 (Phase 3.5):
- **"Single-file archive"**: 케이스 처리 후 `case-report.md` 하나로 전체 협업 과정을 남기는 공유 가능한 아티팩트. 웹 앱보다 단순하지만 GitHub 렌더링만으로 전달력이 높다. → 채택, Phase 3에서 구현.
- **핵심 인사이트**: 프로세스 자체가 프로덕트. 법률 정확도는 table stakes이고, **프로세스의 투명성**(어떤 근거로 결론에 도달했는지 보여주는 것)이 메인 코스. "시각화"는 기능을 깎는 화려한 데모가 아니라, 투명성을 전달하는 수단.
- **AG-UI + CopilotKit**: 조사 가치 있으나 오버엔지니어링 위험. → 보류, 커스텀 정적 뷰어로 시작.
